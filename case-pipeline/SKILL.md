# /case-pipeline — 사례관리 멀티에이전트 파이프라인 스킬

> 역할: 파이프라인 오케스트레이터
> 유형: 멀티에이전트 파이프라인 (Multi-Agent Pipeline)

## 개요

사례관리의 전체 워크플로(접수→사정→개입계획→평가)를 단일 파이프라인으로 실행합니다.
4개의 전문 에이전트가 순차적으로 작동하며, 각 단계의 산출물이 다음 단계의 입력으로 자동 연결됩니다.
SalesLens의 멀티에이전트 파이프라인 패턴에서 영감을 받아, 사례관리에 특화했습니다.

**기존 개별 스킬과의 차이점**: `/intake`, `/assessment`, `/plan-case`, `/evaluate`를 각각 실행하는 대신, `/case-pipeline`을 한 번 실행하면 전체 흐름을 연속적으로 처리합니다. 각 단계 사이에 실천가의 확인과 수정을 받습니다.

## 실행 조건

- 새로운 사례를 처음부터 끝까지 빠르게 처리할 때 사용합니다
- 중간에 언제든 중단하고 개별 스킬로 전환할 수 있습니다
- 각 단계 완료 시 실천가의 확인을 반드시 받습니다

## 파이프라인 구조

```
┌──────────────────────────────────────────────────────────────────┐
│                    /case-pipeline 오케스트레이터                    │
│                                                                  │
│  [Stage 1]         [Stage 2]         [Stage 3]         [Stage 4] │
│  사례접수           사정               개입계획           평가      │
│  에이전트           에이전트           에이전트           에이전트   │
│                                                                  │
│  intake.md ──────→ assessment.md ──→ plan.md ──────→ evaluation.md│
│       │                 │                │                │      │
│       ▼                 ▼                ▼                ▼      │
│  [실천가 확인]     [실천가 확인]    [실천가 확인]     [실천가 확인]  │
└──────────────────────────────────────────────────────────────────┘
```

## 워크플로

### 파이프라인 시작

사용자에게 다음을 질문합니다:

1. **실행 범위**: 전체 파이프라인 / 특정 단계부터 시작
2. **기존 사례 여부**: 신규 사례 / 기존 사례 계속 (case-id 입력)
3. **상세 모드**: 상세(각 단계에서 모든 질문) / 간소화(핵심 질문만)

---

### Stage 1: 사례접수 에이전트

**역할**: 초기 접수상담을 구조화합니다.

**진행 표시**:
```
━━━━━━━━━━ [Stage 1/4: 사례접수] ━━━━━━━━━━
```

**수행 내용**:
1. 의뢰 경로, 클라이언트 기본 정보 수집
2. 주 호소 문제 기록
3. 긴급도 분류 (위기/긴급/일반)
4. 동의서 체크리스트 확인
5. `intake.md` 생성

**위기 감지 시**: 파이프라인을 즉시 중단하고 위기개입 프로토콜 안내
```
⚠️ [파이프라인 중단] 위기 상황이 감지되었습니다.
감지 내용: {구체적 내용}
긴급연락처: 자살예방 1393 / 아동학대 112 / 여성긴급 1366
파이프라인을 위기 대응 후 재개하시겠습니까?
```

**단계 완료 확인**:
```
✅ Stage 1 완료: 접수기록서가 생성되었습니다.
   - 사례번호: {case-id}
   - 긴급도: {등급}
   - 주 호소: {요약}

   다음 단계(Stage 2: 사정)로 진행하시겠습니까?
   [진행] / [수정] / [중단하고 나중에 계속]
```

---

### Stage 2: 사정 에이전트

**역할**: 6개 영역 욕구사정, 강점사정, 위험요인 스크리닝을 수행합니다.

**진행 표시**:
```
━━━━━━━━━━ [Stage 2/4: 사정] ━━━━━━━━━━
📋 접수기록서(intake.md)를 읽고 있습니다...
```

**수행 내용**:
1. `intake.md` 자동 참조
2. 6개 영역별 사정 인터뷰 (신체, 심리정서, 경제, 사회관계, 주거환경, 직업교육)
3. 각 영역에서 욕구/문제와 강점/자원을 동시 파악
4. 표준화 척도 활용 (PHQ-9, GAD-7 등)
5. 생태도 작성
6. 욕구 우선순위 매트릭스 생성
7. 강점 사정
8. 위험요인 스크리닝
9. `assessment.md` 생성

**위험 점수 산출**:
```
위험 수준 종합 평가:
├── 자살/자해: {없음/낮음/중간/높음}
├── 학대/방임: {없음/낮음/중간/높음}
├── 주거 상실: {없음/낮음/중간/높음}
├── 건강 위기: {없음/낮음/중간/높음}
└── 종합 위험도: {낮음/중간/높음}
```

**단계 완료 확인**:
```
✅ Stage 2 완료: 사정보고서가 생성되었습니다.
   - 사례유형: {집중/일반}
   - 핵심 욕구: {목록}
   - 핵심 강점: {목록}
   - 위험 수준: {등급}

   다음 단계(Stage 3: 개입계획)로 진행하시겠습니까?
   [진행] / [수정] / [중단하고 나중에 계속]
```

---

### Stage 3: 개입계획 에이전트

**역할**: 사정 결과를 기반으로 SMART 목표와 서비스 연계 계획을 수립합니다.

**진행 표시**:
```
━━━━━━━━━━ [Stage 3/4: 개입계획] ━━━━━━━━━━
📋 사정보고서(assessment.md)를 읽고 있습니다...
```

**수행 내용**:
1. `assessment.md` 자동 참조 (욕구 우선순위, 강점, 위험요인)
2. SMART 목표 수립 (장기목표 + 단기목표 + 활동)
3. 서비스 연계 계획 (직접서비스/연계/의뢰)
4. 역할 분담 (클라이언트, 담당자, 연계기관)
5. 모니터링 계획
6. 위기 대응 계획
7. 클라이언트 동의 확인
8. `plan.md` 생성

**단계 완료 확인**:
```
✅ Stage 3 완료: 개입계획서가 생성되었습니다.
   - 장기목표: {요약}
   - 단기목표: {N}개
   - 연계기관: {목록}
   - 첫 모니터링: {날짜}

   다음 단계(Stage 4: 평가)로 진행하시겠습니까?
   [진행] / [건너뛰기 — 개입 진행 후 나중에 평가] / [수정]
```

---

### Stage 4: 평가 에이전트

**역할**: 서비스 효과성을 평가하고 종결/연장/재사정을 판단합니다.

**진행 표시**:
```
━━━━━━━━━━ [Stage 4/4: 평가] ━━━━━━━━━━
📋 개입계획서(plan.md)와 상담기록(notes/)을 읽고 있습니다...
```

**참고**: 이 단계는 개입이 진행된 후 실행하는 것이 일반적입니다. 파이프라인 시작 시 Stage 3까지만 실행하고, 개입 진행 후 나중에 Stage 4를 실행할 수 있습니다.

**수행 내용**:
1. `plan.md`와 `notes/` 자동 참조
2. 사후 척도 점수 수집
3. GAS(목표달성척도) 평가
4. 사전-사후 비교 분석
5. 종합 판정 (종결/연장/재사정/의뢰)
6. `evaluation.md` 생성

**단계 완료 확인**:
```
✅ Stage 4 완료: 평가보고서가 생성되었습니다.
   - 목표달성률: {%}
   - 척도 변화: {요약}
   - 판정: {종결/연장/재사정/의뢰}

   ━━━━━━━━━━ [파이프라인 완료] ━━━━━━━━━━
```

## 파이프라인 완료 요약

```markdown
# 사례관리 파이프라인 완료 보고

## 사례 개요
- 사례번호: {case-id}
- 파이프라인 시작: {날짜}
- 파이프라인 완료: {날짜}

## 단계별 요약

| 단계 | 완료일 | 핵심 산출물 | 주요 결과 |
|------|--------|-----------|----------|
| 1. 접수 | {날짜} | intake.md | 긴급도: {등급} |
| 2. 사정 | {날짜} | assessment.md | 사례유형: {유형}, 위험도: {등급} |
| 3. 계획 | {날짜} | plan.md | 목표 {N}개, 연계기관 {N}곳 |
| 4. 평가 | {날짜} | evaluation.md | 달성률: {%}, 판정: {결과} |

## 생성된 파일
- `.swstack/cases/{case-id}/intake.md`
- `.swstack/cases/{case-id}/assessment.md`
- `.swstack/cases/{case-id}/plan.md`
- `.swstack/cases/{case-id}/evaluation.md`

## 다음 단계
{판정 결과에 따른 후속 조치}
```

## 산출물 저장 경로

각 단계의 산출물은 기존 개별 스킬과 동일한 경로에 저장됩니다:
- `.swstack/cases/{case-id}/intake.md`
- `.swstack/cases/{case-id}/assessment.md`
- `.swstack/cases/{case-id}/plan.md`
- `.swstack/cases/{case-id}/evaluation.md`

파이프라인 실행 로그는 `.swstack/cases/{case-id}/pipeline-log.md`에 저장합니다.

## 개인정보 보호 규칙

- 각 단계에서 개별 스킬과 동일한 개인정보 보호 원칙을 적용합니다
- 파이프라인 전체에서 `/privacy-guard`가 암묵적으로 활성화됩니다
- 위기 상황 감지 시 즉시 실천가에게 알립니다

## 사용 예시

```
사용자: /case-pipeline
AI: 사례관리 파이프라인을 시작합니다.

    실행 범위를 선택해주세요:
    1) 전체 파이프라인 (접수→사정→계획→평가)
    2) 특정 단계부터 시작 (기존 사례)

사용자: 전체 파이프라인, 신규 사례

AI: ━━━━━━━━━━ [Stage 1/4: 사례접수] ━━━━━━━━━━
    접수 에이전트를 시작합니다.
    의뢰 경로를 알려주세요.

사용자: 주민센터 의뢰, 70대 여성, 독거, 낙상 후 거동 불편

AI: 접수 정보를 정리하겠습니다.
    ...
    ✅ Stage 1 완료: 접수기록서가 생성되었습니다.
       - 사례번호: CASE-20260329-001
       - 긴급도: 긴급
       다음 단계(Stage 2: 사정)로 진행하시겠습니까?

사용자: 진행

AI: ━━━━━━━━━━ [Stage 2/4: 사정] ━━━━━━━━━━
    접수기록서를 참조하여 사정을 시작합니다.
    신체적 영역부터 살펴보겠습니다.
    ...
```

## 연결 스킬

- **포함**: `/intake` — Stage 1에서 활용
- **포함**: `/assessment` — Stage 2에서 활용
- **포함**: `/plan-case` — Stage 3에서 활용
- **포함**: `/evaluate` — Stage 4에서 활용
- **보조**: `/case-conference` — 복잡 사례 시 파이프라인 중간에 사례회의 삽입
- **보조**: `/resource-map` — Stage 3에서 자원 연계 시 참고
- **보호**: `/privacy-guard` — 전체 파이프라인에서 암묵적 활성화
- **보호**: `/law-check` — 위기 상황 감지 시 법적 의무 확인
